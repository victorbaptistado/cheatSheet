Debugging & Testing

Debugging

  • Is your code throwing errors? If so, read the error stack trace for the type, description, and location of the error. Go to the error’s location and try to fix.
  • Is your code broken but not throwing errors? Walk through your code using console.log() statements. When unexpected results occur, isolate the bug and try to fix it.
  • Did you locate the bug using steps 1 and 2, but can’t fix the bug? Consult documentation to make sure you are using all JavaScript functionality properly. If you are still stuck, Google your issue and consult Stack Overflow for help. Read solutions or post your own Stack Overflow question if none exist on the topic.

Testing

  • Testing can be overwhelming, because there are so many different types of tests: acceptance tests, integration tests, and unit tests, not to mention test driven development vs. ad hoc testing.

Testing plan

  • Creating a testing plan:
    • The happy path: For the classic Tic Tac Toe example, the happy path is starting with an empty board and playing a game until X wins.
    • Variations on the happy path: A variation on the happy path would be if O wins, or if there is a tie game.
    • Edge cases: An edge case would be if a player tries to play a move in the same square more than once.
    • Complex things: The algorithm to find the winner is the most complex part of this example. Here’s a sample test plan:
      • Test that the initial state of the board is correct (i.e. board is visible and empty)
      • Test that a move can be played
      • Test that moves alternate between X and O
      • Test that a move can be played to a square only once
      • Test that a winner can be found in a row
      • Test that a winner can be found in a column
      • Test that a winner can be found in a diagonal
      • Test that a draw can be found
